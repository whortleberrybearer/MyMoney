## Context

The app already has file selection (open/create), path persistence in `localStorage`, and startup routing in `AppProvider`. The transition to the `dashboard` screen happens as soon as a file is found to exist — no DB initialisation currently occurs.

`@tauri-apps/plugin-sql` (v2) and `drizzle-orm` (v0.45) are already installed. `drizzle-kit` is available for generating migration SQL. The Drizzle schema file (`src/lib/db/schema.ts`) is currently a placeholder.

No migration infrastructure exists yet. The `institution` table and all reference tables need to be defined in the Drizzle schema before migrations can be generated.

## Goals / Non-Goals

**Goals:**
- Define all reference table schemas in Drizzle
- Generate Drizzle migration SQL covering those tables
- Run migrations automatically when a data file is opened (both from startup routing and from explicit open)
- Surface a clear, user-visible error if migration fails; do not swallow errors
- Seed all reference tables idempotently as part of the migration (INSERT OR IGNORE)

**Non-Goals:**
- Creating a new data file from scratch (out of scope per issue)
- Any UI for browsing or editing reference data
- Drizzle migrations for tables beyond the reference set listed in the issue

## Decisions

### 1. Run migrations from the frontend (TypeScript), not from Rust

`tauri-plugin-sql` exposes a JavaScript API. Drizzle's migrate function for SQLite works with an abstract driver interface. Rather than duplicating schema knowledge in Rust, we use Drizzle's `migrate()` from the frontend with a thin adapter that proxies SQL execution through `tauri-plugin-sql`.

**Alternative considered**: Rust-side migration using `rusqlite` or `sqlx`. Rejected because it would require duplicating schema definitions and adding non-trivial Rust dependencies, while the JS layer already has full access to the DB.

### 2. Centralized DB singleton in `src/lib/db/index.ts`

A module-level singleton holds the open Drizzle instance. It exposes two functions:
- `openDb(filePath: string): Promise<void>` — opens the SQLite connection and runs `migrate()`. Throws on failure.
- `getDb(): DrizzleInstance` — returns the open instance. Throws if not initialized.

This keeps all DB setup in one place and makes it easy to call from both startup routing and the explicit file-open flow.

**Alternative considered**: Storing the DB connection in React context. Rejected because DB access will be needed outside of React (utility functions, etc.), and a module singleton is simpler and avoids prop-drilling.

### 3. Migration SQL generated by drizzle-kit, seeding included in migration

Reference table seed data is inserted within a dedicated migration file using `INSERT OR IGNORE INTO … VALUES …`. This ensures:
- Seed runs automatically as part of `migrate()`
- Idempotency is guaranteed by SQLite's `OR IGNORE` conflict resolution on the primary key
- No separate seed step or runtime logic needed

**Alternative considered**: A separate `runSeed()` call after `migrate()`. Rejected because it adds complexity and a separate failure mode. Seed data is stable and belongs to the schema lifecycle.

### 4. Migration error surfaces as a new `AppScreen` state

A new screen state `{ screen: "migration-error"; filePath: string; error: string }` is added to `AppScreen`. The startup routing and file-open helpers catch migration errors and navigate to this screen instead of `dashboard`. The migration-error screen shows the error message and lets the user either retry or go back to Welcome.

**Alternative considered**: A toast or modal dialog. Rejected because a migration failure is a blocking error — the user cannot use the dashboard without a valid DB. A dedicated screen makes this clear and avoids partial UI rendering.

### 5. Drizzle SQLite adapter via `tauri-plugin-sql`

Drizzle's SQLite dialect requires a driver that can execute SQL and return rows. `tauri-plugin-sql` exposes `Database.execute()` and `Database.select()`. We write a minimal adapter object conforming to Drizzle's `SQLiteAdapter` interface that delegates to these methods.

## Risks / Trade-offs

- **drizzle-kit migration generation requires a running schema**: The `schema.ts` file must be complete before `drizzle-kit generate` can produce SQL. If the schema is wrong, regeneration is needed. Mitigation: keep schema.ts as the single source of truth; never hand-edit migration SQL.
- **`INSERT OR IGNORE` relies on primary key uniqueness**: Seed rows must have a stable identity column (integer PK or unique name). All reference tables in scope have a suitable PK. No risk.
- **tauri-plugin-sql adapter is hand-rolled**: Drizzle does not ship an official tauri-plugin-sql adapter. The adapter is small (~30 lines) and low-risk, but any Drizzle upgrade could change the driver interface. Mitigation: keep the adapter isolated in `src/lib/db/adapter.ts` so it's easy to update.
- **Migration runs on every file open**: `migrate()` is a no-op if the schema is already current, so repeated opens are safe. Slight startup latency is acceptable for a desktop app.

## Migration Plan

1. Define reference table schemas in `src/lib/db/schema.ts`
2. Run `drizzle-kit generate` to produce migration SQL in `src/lib/db/migrations/`
3. Edit the generated migration to append `INSERT OR IGNORE` seed statements
4. Implement the tauri-plugin-sql adapter (`src/lib/db/adapter.ts`)
5. Implement `openDb` / `getDb` in `src/lib/db/index.ts`
6. Update `useStartupRouting` in `app-context.tsx` to call `openDb` before navigating to `dashboard`
7. Update `openExistingFile` in `file-selection.ts` to call `openDb` after selecting a file
8. Add `migration-error` screen to `AppScreen` union and render a basic error screen component
9. Wire the migration-error screen into `App.tsx`

No rollback strategy needed — this is additive schema work on local SQLite files owned entirely by the user.
